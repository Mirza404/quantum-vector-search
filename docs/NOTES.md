## Problem Statement

**Cross-modal similarity search:** text queries return matching images via shared vector embeddings. We benchmark classical engines against quantum engines to measure:
- **Accuracy** (MRR)
- **Quantum resource cost** (circuit depth, qubits, shots)

We do **not** compare raw speed -- the quantum engine runs on a classical simulator.

> **Analogy:** We're taste-testing dishes from two kitchens (classical and quantum). We care about taste quality (MRR) and ingredient cost (qubits, depth), not cooking time (because one kitchen is using an oven simulator, not a real oven).

## Engineering Philosophy

| Principle | What it means | Where you see it |
|---|---|---|
| **Modular Monolith** | Single repo, isolated folders. Web server is agnostic to quantum math | `backend/`, `frontend/`, `db/` directories |
| **API-First** | Logic behind FastAPI. Frontend requires zero backend changes | `backend/app/` |
| **Strategy Pattern** | Engines, embedders, data loaders share abstract interfaces | `base.py` in each `src/` subpackage |
| **Config-Driven** | YAML controls experiments, not code changes | `backend/config/benchmarks.yaml` |

## Implementation Phases

### Phase 1: Repository Layer

- **Interface:** `BaseDataLoader` in `backend/src/repository/base.py`
- **Implementation:** `DirectoryDataLoader` in `backend/src/repository/local_csv.py` -- reads WebP images from `backend/data/images/`, uses filename stem as record ID
- **Ground truth:** `backend/data/ground_truth.jsonc`, generated by `scripts/import_dataset.py` (pulls from Flickr30k via HuggingFace dataset viewer API)
- **Config:** `backend/config/dataset.yaml` controls `num_images` (default: 20)

### Phase 2: Embedding Pipeline

- **Interface:** `EmbeddingGenerator` in `backend/src/pipeline/base.py`
- **Real implementation:** `CLIPEmbeddingModel` + `CLIPTextEmbeddingGenerator` in `backend/src/pipeline/clip_model.py`
  - ViT-B/32, auto-detects CUDA/MPS/CPU
  - `encode_texts()` for text, `encode_images()` for images
  - Both output into shared 512-dim space
- **Test implementation:** `MockCLIPEmbeddingGenerator` in `backend/src/pipeline/mock_clip.py` -- deterministic pseudo-random vectors via SHA-256

### Phase 3: Search Engines

Four engines implementing `SearchEngineStrategy` (`build_index()` + `search()` in `backend/src/engines/base.py`):

| Engine name | Class | File | What it does |
|---|---|---|---|
| `brute_force_cosine` | `BruteForceCosineEngine` | `brute_force_cosine.py` | NumPy dot products. Exact. **Ground truth** |
| `faiss_flat_l2` | `FaissFlatEngine` | `faiss_flat.py` | FAISS IndexFlatL2. Exact. Production-grade |
| `quantum_mock_sampler` | `QuantumMockEngine` | `quantum_mock.py` | Cosine + noise (stdev = layers/shots). No circuits |
| `qiskit_swap_test` | `QiskitSwapTestEngine` | `qiskit_swaptest.py` | Real swap test on AerSimulator |

All files in `backend/src/engines/`. Controlled by `backend/config/benchmarks.yaml` -- comment out entries to skip.

### Phase 4: Database Storage

All results write to PostgreSQL via `DatabaseStorage` in `backend/src/benchmark/db_storage.py`.

**Infrastructure (in `db/`):**

| Component | File | Purpose |
|---|---|---|
| Database | `docker-compose.yml` | `pgvector/pgvector:pg16` + Adminer (localhost:8080) |
| Migrations | `migrations/up/*.sql` | Numbered SQL files, applied by `migrate.sh up` |
| Rollback | `migrations/down/*.sql` | Paired rollback scripts |
| Seed data | `seeds/seed.sql` | Data snapshot. `make seed` resets to this state |

**Workflow:**
```bash
cd db
make up        # start Postgres
make seed      # load seed data + apply migrations
make dump      # export current data to seed.sql (then commit)
```

**`benchmark_results` table (after all migrations):**

| Column | Type | Notes |
|---|---|---|
| `query_id` | TEXT | e.g. "query_1000092795" |
| `engine_name` | TEXT | e.g. "brute_force_cosine" |
| `dimension` | INTEGER | Vector dimension used |
| `shots` | INTEGER | -1 for classical engines |
| `layers` | INTEGER | -1 for classical engines |
| `target_ids` | JSONB | Correct image IDs |
| `top_ids` | JSONB | Ranked results |
| `state_prep_ms` | FLOAT | Index build time |
| `search_ms` | FLOAT | Search time |
| `total_ms` | FLOAT | state_prep + search |
| `circuit_depth` | INTEGER | NULL for classical |
| `num_qubits` | INTEGER | NULL for classical |

**Run key:** `(query_id, engine_name, dimension, shots, layers)` -- unique constraint (`uq_run_key`). Re-running upserts; new params append.

**`image_vectors` table:** `id` TEXT PK, `embedding` vector(512), `recorded_at`. Has HNSW cosine index for future live search.

### Phase 5: API & React Dashboard (in progress)

- **FastAPI:** `backend/app/main.py` creates the app. `backend/app/api/router.py` has a stub router at `/api`. Planned: `GET /search?q=<text>` -> CLIP encode -> pgvector kNN -> top-K results
- **React:** `frontend/` -- Vite + React + TypeScript. Currently shows the default Vite starter. Planned: search bar, image grid, benchmark dashboard
- **What works now:** `scripts/index_dataset.py` encodes images and stores vectors in `image_vectors`. What remains: FastAPI search endpoint and React wiring

## Configuration

`backend/config/benchmarks.yaml` controls each benchmark run:

| Section | What it controls | Example |
|---|---|---|
| `classical_engines` | Which classical engines run | `brute_force_cosine`, `faiss_flat_l2` |
| `quantum_engines` | Which quantum engines run | `quantum_mock_sampler`, `qiskit_swap_test` |
| `dimensions` | Vector sizes to test | 64, 128, 256, 512 |
| `shots_values` | Shots per comparison (quantum only) | 512, 1024, 2048 |
| `layers_values` | Gate layers (mock only) | 1, 2, 3 |

CLI flags override YAML for one-off runs: `--dimensions`, `--shots-values`, `--layers-values`, `--clip-model`, `--device`, `--batch-size`.

## KPIs & Metrics

### Quality (cross-engine comparison)

| Metric | How it's computed | Code location |
|---|---|---|
| **MRR** | 1 / rank of first correct result, averaged over all queries | `_mrr()` in `run_benchmarks.py` |

Full ranking (all images), no top-K cutoff. Best engine = highest MRR.

### Quantum Resource Cost

| Metric | What it measures | Why it matters |
|---|---|---|
| **Circuit depth** | Sequential gate layers | Decoherence risk. NISQ limit: ~100 |
| **Qubit count** | Qubits per swap test | Hardware allocation. Current chips: 100-1000 |
| **Shots vs. MRR** | Quality at each measurement budget | Cost parameter on real hardware |

### Speed

Valid only **within one engine** across dimensions (e.g. dim=64 vs dim=128 for the same engine). **Not valid across engines** -- the quantum engine's timing reflects classical simulation overhead.

**In one sentence:** MRR is the single quality metric for cross-engine comparison; circuit depth, qubits, and shots are quantum-specific cost metrics; speed is only meaningful within a single engine.

<details>
<summary>Self-test</summary>

**Q: Why doesn't the project compare speed across engines?**
A: Because the quantum engine runs on a classical simulator. Its wall-clock time reflects simulation cost, not actual quantum hardware speed.

**Q: What's the run key for benchmark results, and why does it matter?**
A: `(query_id, engine_name, dimension, shots, layers)`. This ensures each configuration has exactly one row. Re-running the same config overwrites (upsert), new configs append.

**Q: Where are CLIP embeddings stored, and how are they used during benchmarking?**
A: In the `image_vectors` table (stored by `scripts/index_dataset.py`). During benchmarking, `run_benchmarks.py` loads them via `storage.load_image_vectors()`, truncates to the configured dimension, and feeds them to each engine.
</details>
