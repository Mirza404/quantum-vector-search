## Problem Statement

Cross-modal similarity search: text queries return matching images via shared vector embeddings.
We benchmark classical engines against quantum engines to measure accuracy (MRR) and quantum
resource cost (circuit depth, qubits, shots). We do not compare raw speed — quantum runs on a
classical simulator.

## Objectives & Deliverables

**Objectives:**
- Parallel classical and quantum search engines, comparable side-by-side.
- Shared CLIP embedding pipeline for text and images.
- Interactive web interface for text queries → matching images.
- Analysis and conclusions from benchmark metrics.

**Deliverables:**
- Working web application with cross-modal search.
- Experimental dataset and evaluation pipeline.
- Classical vs. quantum comparison with conclusions.

## Engineering Philosophy

- **Modular Monolith:** Single repo, isolated folders. Web server is agnostic to quantum math.
- **API-First:** Logic behind FastAPI. React frontend requires zero backend changes.
- **No UI in MVP:** Driven by automated Python benchmarking scripts.
- **Strategy Pattern:** Engines, embedders, and data loaders share abstract interfaces.

## Implementation Phases

### Phase 1: Repository Layer
`BaseDataLoader` interface → `DirectoryDataLoader` reads WebP images from `backend/data/images/`.
Ground truth from `backend/data/ground_truth.jsonc`, generated by `scripts/import_dataset.py`
(Flickr30k via HuggingFace). No user uploads in MVP.

### Phase 2: Embedding Pipeline
`EmbeddingGenerator` interface → `CLIPEmbeddingModel` (ViT-B/32) and `MockCLIPEmbeddingGenerator`
for tests. Encodes images and text into shared 512-dim space on each benchmark run.

### Phase 3: Search Engines
Four engines implementing `SearchEngineStrategy` (`build_index()` + `search()`):

| Engine | Class | Description |
|---|---|---|
| `brute_force_cosine` | `BruteForceCosineEngine` | Deterministic cosine, NumPy. Ground truth baseline. |
| `faiss_flat_l2` | `FaissFlatEngine` | FAISS `IndexFlatL2`. Production-grade classical. |
| `quantum_mock_sampler` | `QuantumMockEngine` | Cosine + configurable noise. No real circuits. |
| `qiskit_swap_test` | `QiskitSwapTestEngine` | Real swap test on AerSimulator. |

Controlled by `backend/config/benchmarks.yaml` — comment out entries to skip.

### Phase 4: Database Storage
All results write to PostgreSQL. No CSV files.

**Infrastructure (in `db/`):**
- `docker-compose.yml` — `pgvector/pgvector:pg16` + Adminer (http://localhost:8080).
- `migrations/up/` — numbered SQL files applied by `migrate.sh up`.
- `migrations/down/` — paired rollback scripts.
- `seeds/seed.sql` — data snapshot. `make seed` resets DB to this state. `make dump` updates it.

**Workflow:**
```bash
cd db
make up        # start Postgres
make seed      # load seed data + apply migrations
make dump      # export current data to seed.sql (then commit)
```

**`benchmark_results` columns:**
`query_id`, `engine_name`, `dimension`, `shots`, `layers`, `target_ids`, `top_ids`,
`state_prep_ms`, `search_ms`, `total_ms`, `parameters`, `dataset_size`, `circuit_depth`,
`num_qubits`

Run key: `(query_id, engine_name, dimension, shots, layers)`. Classical engines use
`shots = -1, layers = -1`. Upsert on conflict.

**`image_vectors`:** `id` TEXT PK, `embedding` vector(512), `recorded_at`. HNSW cosine index.

### Phase 5: API & React Dashboard (in progress)
- **FastAPI:** `GET /search?q=<text>` → CLIP encode → pgvector nearest-neighbour → top-K IDs.
  `GET /api/benchmarks` → historical results.
- **React:** Search bar, image grid, benchmark dashboard.
- Vector storage is implemented (`scripts/index_dataset.py`). What remains: FastAPI endpoints
  and React wiring.

## Configuration

`backend/config/benchmarks.yaml` controls each run:
- `classical_engines`, `quantum_engines`, `dimensions` — comment out to skip.
- `shots_values`, `layers_values` — each combination produces a separate DB row.

CLI flags (`--shots-values`, `--layers-values`, `--dimensions`) override YAML for one-off runs.

## KPIs & Metrics

Cross-engine comparisons use **quality only**. No cross-engine speed comparison.

**Quality (cross-engine):**

| Metric | Description |
|---|---|
| MRR | 1 / rank of first correct result, averaged over all queries. Full ranking, no top-K cutoff. |

Best engine = highest average MRR. Ties labelled as such.

**Quantum resource cost:**

| Metric | Description | Thesis argument |
|---|---|---|
| Circuit depth | Sequential gate layers | Decoherence risk proxy — feasibility on near-term hardware |
| Num qubits | Qubits per swap test | Hardware allocation requirement |
| Shots vs. MRR | Measurement budget for target accuracy | Cost parameter for real hardware deployment |

**Speed:** Valid only within one engine across dimensions (dim=64 vs dim=128). Not valid
across engines.
