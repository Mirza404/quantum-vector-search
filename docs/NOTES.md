## What This Project Does

You type "a dog on a beach". The app finds matching images - even though nobody labelled them. Both text and images are converted into 512-number vectors by CLIP, and the system finds the closest ones.

The twist: **how** the matching is done. A classical engine uses CPU math. A quantum engine uses quantum circuits. We benchmark both to measure whether the quantum approach works and what it costs.

> Two GPS systems finding the nearest restaurant. One uses Google Maps (classical). One uses a prototype quantum GPS. We test whether the quantum GPS gives correct directions, not whether it's faster.

We do **not** compare raw speed - the quantum engine runs on a classical simulator.

## Engineering Principles

| Principle | Where you see it |
|---|---|
| **Modular Monolith** - single repo, isolated folders | `backend/`, `frontend/`, `db/` |
| **API-First** - logic behind FastAPI | `backend/app/main.py` + `backend/app/api/router.py` |
| **Strategy Pattern** - engines, embedders, loaders share abstract interfaces | `base.py` in each `src/` subpackage |
| **Config-Driven** - YAML controls experiments, not code changes | `backend/config/benchmarks.yaml` |

## Implementation Phases

### Phase 1: Data Loading

- **Interface:** `BaseDataLoader` in `backend/src/repository/base.py`
- **Implementation:** `DirectoryDataLoader` in `local_csv.py` - reads WebP/JPG/PNG images from `backend/data/images/`, filename stem = record ID
- **Ground truth:** `backend/data/ground_truth.jsonc` - 20 image-text pairs from Flickr30k, generated by `scripts/import_dataset.py` (pulls images via HuggingFace API)
- **Config:** `backend/config/dataset.yaml` - `num_images: 20`

### Phase 2: Embedding Pipeline

- **Interface:** `EmbeddingGenerator` in `backend/src/pipeline/base.py`
- **Real:** `CLIPEmbeddingModel` in `clip_model.py` - ViT-B/32, auto-detects CUDA/MPS/CPU, L2-normalises output. Runs on CPU without a GPU.

### Phase 3: Search Engines

Four engines implementing `SearchEngineStrategy` (`build_index()` + `search()`) in `backend/src/engines/base.py`:

| Engine | Class | File | Purpose |
|---|---|---|---|
| `brute_force_cosine` | `BruteForceCosineEngine` | `brute_force_cosine.py` | NumPy dot products. Exact. **Ground truth** |
| `faiss_flat_l2` | `FaissFlatEngine` | `faiss_flat.py` | FAISS L2. Exact. Production-grade |
| `qiskit_swap_test` | `QiskitSwapTestEngine` | `qiskit_swaptest.py` | Real swap test on AerSimulator |
| `qiskit_grover` | `QiskitGroverEngine` | `qiskit_grover.py` | Grover's algorithm. O(sqrt(N)) oracle scaling |

Controlled by `benchmarks.yaml` - comment out entries to skip.

### Phase 4: Database Storage

Results go to PostgreSQL via `DatabaseStorage` in `backend/src/benchmark/db_storage.py`.

**Infrastructure (`db/`):**
- `docker-compose.yml` - `pgvector/pgvector:pg16` on port 6432 + Adminer on port 8080
- `migrations/up/*.sql` - schema evolution (5 migrations)
- **Workflow:** `make up` -> `make seed` -> run benchmarks -> `make dump`

**`benchmark_results` table (key columns):**

| Column | Type | Notes |
|---|---|---|
| `query_id` | TEXT | e.g. `query_1000092795` |
| `engine_name` | TEXT | e.g. `brute_force_cosine` |
| `dimension` | INTEGER | Vector dimension used |
| `shots` / `layers` | INTEGER | -1 for classical engines |
| `target_ids` / `top_ids` | JSONB | Ground truth / ranked results |
| `search_ms` / `state_prep_ms` / `total_ms` | FLOAT | Timing |
| `circuit_depth` / `num_qubits` | INTEGER | NULL for classical |
| `oracle_calls` | INTEGER | N for classical, floor(pi*sqrt(N)/4) for Grover |

**Run key:** `(query_id, engine_name, dimension, shots, layers)` - unique constraint. Same config = upsert, new config = append.

**`image_vectors` table:** stores CLIP embeddings as `vector(512)` with an HNSW cosine index.

### Phase 5: API & Frontend

- **FastAPI:** `backend/app/main.py` creates the app with CORS for Vite dev server. `router.py` provides:
  - `GET /api/images` - paginated image listing
  - `GET /api/images/{id}` - serve image file
  - `GET /api/queries` - list ground-truth queries
  - `GET /api/search?query_id=...` - runs both classical and quantum engines side-by-side, returns MRR and ranked results for each
- **React frontend:** `frontend/` - Vite + React + TypeScript
  - `App.tsx` - tab navigation (Browse / Search)
  - `ImageBrowser.tsx` - paginated image grid
  - `QueryPicker.tsx` - dropdown to select a query
  - `SearchResults.tsx` - side-by-side comparison showing MRR, target rank, and search_ms per engine

## Configuration

`backend/config/benchmarks.yaml` controls each benchmark run:

| Section | Controls | Example values |
|---|---|---|
| `classical_engines` | Which classical engines run | `brute_force_cosine`, `faiss_flat_l2` |
| `quantum_engines` | Which quantum engines run | `qiskit_swap_test`, `qiskit_grover` |
| `dimensions` | Vector sizes to test | 64, 128, 256, 512 |
| `shots_values` | Shots per comparison (quantum) | 512, 1024, 2048 |
| `layers_values` | Required by harness, ignored by current engines | 1 |
| `top_k` | Results per search query | 10 |

CLI flags override YAML: `--dimensions`, `--shots-values`, `--layers-values`, `--clip-model`, `--device`, `--batch-size`.

Environment variables override YAML for the API: `CLASSICAL_ENGINE`, `QUANTUM_ENGINE`, `SEARCH_DIMENSION`, `SEARCH_SHOTS`, `SEARCH_LAYERS`, `SEARCH_TOP_K`.

## KPIs

- **MRR** (cross-engine): 1/rank of first correct result, averaged. Code: `_mrr()` in `run_benchmarks.py`
- **Operation count** (cross-engine): `oracle_calls` column. N for classical, floor(pi*sqrt(N)/4) for Grover
- **Circuit depth / qubits**: hardware cost proxies (quantum only)
- **Shots vs. MRR**: quality at each measurement budget (quantum only)
- **Wall-clock speed**: valid only **within one engine** across dimensions. Never across engines

See [BENCHMARK_KPIS.md](BENCHMARK_KPIS.md) for the full specification.

**In one sentence:** MRR compares accuracy across engines; operation count compares scaling; depth, qubits, and shots measure quantum cost; wall-clock speed is per-engine only.

**Self-test**

**Q: Why not compare speed across engines?**
A: Quantum engine timing reflects classical simulation cost, not real hardware speed.

**Q: What's the run key?**
A: `(query_id, engine_name, dimension, shots, layers)`. Ensures one row per config combination.

**Q: Where are CLIP embeddings stored?**
A: `image_vectors` table. Stored by `index_dataset.py`, loaded by `run_benchmarks.py` via `storage.load_image_vectors()`.
