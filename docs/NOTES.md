# Project Architecture & Specification

System design, implementation details, and metrics. For theory: [LEARNING_ROADMAP.md](LEARNING_ROADMAP.md). For quick reference: [THEORY.md](THEORY.md).

## Problem Statement

**Cross-modal similarity search:** text queries return matching images via shared vector embeddings. We benchmark classical vs. quantum engines on:
- **Accuracy** (MRR)
- **Quantum resource cost** (circuit depth, qubits, shots)

We do **not** compare raw speed -- the quantum engine runs on a classical simulator.

> **Analogy:** Taste-testing dishes from two kitchens. We care about taste (MRR) and ingredient cost (qubits, depth), not cooking time (one kitchen uses a simulated oven).

---

## Objectives & Deliverables

- Classical and quantum search engines, comparable side-by-side
- Shared CLIP embedding pipeline for text and images
- Interactive web interface for text-to-image search
- Benchmark analysis with conclusions
- Experimental dataset (Flickr30k subset) and evaluation pipeline

## Engineering Philosophy

| Principle | Where you see it |
|---|---|
| **Modular Monolith** -- single repo, isolated folders | `backend/`, `frontend/`, `db/` |
| **API-First** -- logic behind FastAPI | `backend/app/` |
| **Strategy Pattern** -- engines, embedders, loaders share abstract interfaces | `base.py` in each `src/` subpackage |
| **Config-Driven** -- YAML controls experiments, not code changes | `backend/config/benchmarks.yaml` |

## Implementation Phases

### Phase 1: Repository Layer

- **Interface:** `BaseDataLoader` in `backend/src/repository/base.py`
- **Implementation:** `DirectoryDataLoader` in `local_csv.py` -- reads WebP images from `backend/data/images/`, filename stem = record ID
- **Ground truth:** `backend/data/ground_truth.jsonc`, generated by `scripts/import_dataset.py` (pulls Flickr30k via HuggingFace API)
- **Config:** `backend/config/dataset.yaml` -- `num_images` (default: 20)

### Phase 2: Embedding Pipeline

- **Interface:** `EmbeddingGenerator` in `backend/src/pipeline/base.py`
- **Real:** `CLIPEmbeddingModel` + `CLIPTextEmbeddingGenerator` in `clip_model.py` -- ViT-B/32, auto-detects CUDA/MPS/CPU
- **Mock:** `MockCLIPEmbeddingGenerator` in `mock_clip.py` -- deterministic pseudo-random vectors via SHA-256

### Phase 3: Search Engines

Four engines implementing `SearchEngineStrategy` (`build_index()` + `search()`) in `backend/src/engines/base.py`:

| Engine | Class | File | Purpose |
|---|---|---|---|
| `brute_force_cosine` | `BruteForceCosineEngine` | `brute_force_cosine.py` | NumPy dot products. Exact. **Ground truth** |
| `faiss_flat_l2` | `FaissFlatEngine` | `faiss_flat.py` | FAISS L2. Exact. Production-grade |
| `quantum_mock_sampler` | `QuantumMockEngine` | `quantum_mock.py` | Cosine + noise. No circuits |
| `qiskit_swap_test` | `QiskitSwapTestEngine` | `qiskit_swaptest.py` | Real swap test on AerSimulator |

Controlled by `benchmarks.yaml` -- comment out entries to skip.

### Phase 4: Database Storage

Results go to PostgreSQL via `DatabaseStorage` in `backend/src/benchmark/db_storage.py`.

**Infrastructure (`db/`):**

| Component | File |
|---|---|
| Database | `docker-compose.yml` -- `pgvector/pgvector:pg16` + Adminer (port 8080) |
| Migrations | `migrations/up/*.sql`, applied by `migrate.sh up` |
| Seed data | `seeds/seed.sql` -- `make seed` resets to this state |

**Workflow:** `make up` (start Postgres) -> `make seed` (load data + migrate) -> `make dump` (export to seed.sql, then commit)

**`benchmark_results` table:**

| Column | Type | Notes |
|---|---|---|
| `query_id` | TEXT | e.g. "query_1000092795" |
| `engine_name` | TEXT | e.g. "brute_force_cosine" |
| `dimension` | INTEGER | Vector dimension used |
| `shots` | INTEGER | -1 for classical |
| `layers` | INTEGER | -1 for classical |
| `target_ids` / `top_ids` | JSONB | Ground truth / ranked results |
| `state_prep_ms` / `search_ms` / `total_ms` | FLOAT | Timing |
| `circuit_depth` / `num_qubits` | INTEGER | NULL for classical |

**Run key:** `(query_id, engine_name, dimension, shots, layers)` -- unique constraint `uq_run_key`. Re-running upserts; new params append.

**`image_vectors` table:** `id` TEXT PK, `embedding` vector(512), HNSW cosine index for future live search.

### Phase 5: API & React Dashboard (in progress)

- **FastAPI:** `backend/app/main.py` creates the app. `backend/app/api/router.py` has a stub router at `/api`. Planned: `GET /search?q=<text>` -> CLIP encode -> pgvector kNN -> top-K
- **React:** `frontend/` -- Vite + React + TypeScript. Currently default starter. Planned: search bar, image grid, benchmark dashboard
- **What works now:** `scripts/index_dataset.py` stores vectors; what remains is the FastAPI endpoint and React wiring

## Configuration

`backend/config/benchmarks.yaml` controls each run:

| Section | Controls | Example |
|---|---|---|
| `classical_engines` | Which classical engines run | `brute_force_cosine`, `faiss_flat_l2` |
| `quantum_engines` | Which quantum engines run | `quantum_mock_sampler`, `qiskit_swap_test` |
| `dimensions` | Vector sizes to test | 64, 128, 256, 512 |
| `shots_values` | Shots per comparison (quantum) | 512, 1024, 2048 |
| `layers_values` | Gate layers (mock only) | 1, 2, 3 |

CLI flags override YAML: `--dimensions`, `--shots-values`, `--layers-values`, `--clip-model`, `--device`, `--batch-size`.

## KPIs & Metrics

- **MRR** (cross-engine): 1/rank of first correct result, averaged. Full ranking, no top-K cutoff. Code: `_mrr()` in `run_benchmarks.py`
- **Circuit depth**: Sequential gate layers. Decoherence proxy. NISQ limit ~100
- **Qubit count**: Per swap test. Current chips: 100-1000
- **Shots vs. MRR**: Quality at each measurement budget
- **Speed**: Valid only **within one engine** across dimensions. Not valid across engines (simulator overhead)

**In one sentence:** MRR compares accuracy across engines; depth, qubits, and shots measure quantum cost; speed is only meaningful within a single engine.

<details><summary>Self-test</summary>

**Q: Why not compare speed across engines?**
A: Quantum engine timing reflects classical simulation cost, not real hardware speed.

**Q: What's the run key and why?**
A: `(query_id, engine_name, dimension, shots, layers)`. Ensures one row per config. Same config = upsert, new config = append.

**Q: Where are CLIP embeddings stored?**
A: `image_vectors` table (stored by `index_dataset.py`). During benchmarking, `run_benchmarks.py` loads them via `storage.load_image_vectors()`, truncates to configured dimension, feeds to each engine.
</details>
