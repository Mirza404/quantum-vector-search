Context and Motivation

Why this problem matters

The search problem

  • Modern retrieval systems work in vector space -- text, images, audio all reduced to high-dimensional embeddings.
  • Cross-modal search (text query finding images) requires a shared embedding space: different modalities, same geometry.
  • As datasets grow to hundreds of millions of vectors, brute-force scan stops being viable.

Why quantum is worth studying here

  • Grover's algorithm provides a proven, unconditional quadratic speedup for unstructured search -- one of very few such results in quantum computing.
  • Quantum hardware is advancing. Understanding what algorithms require today informs engineering decisions for tomorrow.
  • Empirical benchmarks of quantum search on realistic embeddings are scarce in the literature.

CLIP aligns text and image encoders in a shared vector space.[6]   Source: Radford et al., 2021 (OpenAI)

Problem Definition, Scope, and Objectives

What we are building and measuring

Research questions

  • Does the Grover oracle call count follow the expected O(sqrt(N)) curve empirically across dataset sizes?
  • Can the quantum swap test match classical MRR at a justifiable shot budget?
  • Is quantum vector search worth using even if the hardware problem (qRAM) were solved?

Objectives

  • Implement and verify quantum algorithms (swap test, Grover oracle) on a real circuit simulator.
  • Benchmark all engines on identical data with well-defined, fair KPIs.
  • Expose results through a web interface: user types a query, sees ranked results from all engines side by side.
  • Produce a rigorous, honest analysis of quantum search feasibility -- today and under future hardware assumptions.

Scope

In scope: Simulator-only benchmarks, CLIP embeddings on Flickr30k, web UI for comparison, academic analysis of quantum resource costs and algorithmic limits.
Out of scope: Real quantum hardware (IBM free tier has multi-hour queue waits and significant noise beyond ~7 qubits -- unsuitable for controlled scaling experiments).[9] Production deployment. Datasets beyond the benchmark configuration.

Why simulator and not IBM hardware

  • AerSimulator is noise-free and deterministic -- ideal for measuring oracle count scaling.
  • Shot noise exists identically on simulator and real hardware; our MRR results are directly transferable.
  • IBM free tier: hours of queue wait, significant noise beyond ~7 qubits -- would obscure the algorithmic signal we're measuring.[9]

Theoretical Background

Three building blocks

CLIP -- shared vector space

  • Contrastive pre-training: paired text + image examples, aligned encoders.
  • Result: cosine similarity between a text vector and an image vector measures semantic match.
  • We use ViT-B/32 at 512 dimensions.[6]

Vector similarity search

  • Brute force cosine: O(N) per query, exact.
  • FAISS Flat: same complexity, SIMD-optimised.
  • HNSW (not benchmarked here, but key reference): O(log N) approximate -- the production standard.[5]

Quantum swap test

  • 1 ancilla + two amplitude-encoded registers (each log2(dim) qubits).
  • Hadamard -- controlled-SWAP -- Hadamard -- measure ancilla.
  • P(ancilla = 0) = (1 + |<q|d>|^2) / 2, giving direct overlap estimate.[3]

Grover's algorithm -- O(sqrt(N)) oracle calls

Grover's circuit: oracle marks the answer, diffusion amplifies it.
Source: Wikipedia Commons

  • Oracle: flips the phase of the target state. Diffusion: amplifies the target amplitude toward 1.
  • Optimal iterations: floor(pi * sqrt(N) / 4). Provably optimal for unstructured quantum search.[1][2]
  • Has not been "dequantized" -- Grover's quadratic speedup is real and unconditional.[7]
The O(sqrt(N)) figure counts oracle calls, not total steps. Getting data into superposition (state preparation) is a separate cost -- see slide 10.

Targets, KPIs, and Success Measures

How we evaluate correctness, quality, and cost

Primary quality metric: MRR

Mean Reciprocal Rank = average of (1 / rank of first correct result) across all queries. Comparable across all engines because all use the same normalised vectors and ground truth.

MRRInterpretation
1.0Correct result always ranked 1st
0.5Correct result on average ranked 2nd
0.33Correct result on average ranked 3rd

Quantum resource metrics

  • Shots-to-quality: MRR at each shots value -- minimum measurement budget for acceptable accuracy.
  • Circuit depth: sequential gate layers. NISQ hardware practical limit is roughly 100.
  • Qubit count: 64-dim swap test uses 13 qubits; 128-dim uses 15 qubits. Grows with log2(dim).

  • Frontend comparison view communicates trade-offs to a non-specialist viewer.
  • Results are reproducible: same inputs produce same report on a fresh run.

The Honest Quantum Picture

What the theory and the numbers actually say

The two-step problem

  • Step 1 -- State preparation: Load all N vectors into superposition. Without qRAM this is O(N) gates -- same as classical scan.[4]
  • Step 2 -- Search: Grover finds the answer in O(sqrt(N)) oracle calls. This works and is verified by our circuits.
Without qRAM: O(N) loading + O(sqrt(N)) search = O(N) total. The speedup requires hardware that does not exist.

qRAM hardware reality

The bucket-brigade qRAM model requires O(N) quantum routing nodes as physical hardware.[4] With error correction (~1,000 physical qubits per logical qubit), even a small dataset demands hardware orders of magnitude beyond anything that exists or is on any roadmap:

Dataset sizePhysical qubits needed*IBM Condor (2023) [9]
1,000 vectors~1,000,000~1,100
1,000,000 vectors~1,000,000,000--

* Bucket-brigade model [4] with error correction (~1000 physical per logical qubit). Note: qRAM is a different hardware architecture from processor qubits -- IBM's processor qubits cannot substitute.

Research finding: even ideal qRAM may not be enough

MethodComplexityExact?Requires
Brute forceO(N)YesNothing
HNSW [5]O(log N)Approximate*Nothing
Grover + ideal qRAM [1][4]O(sqrt(N))YesqRAM (does not exist)

* HNSW recall is 95-99%+ in practice -- the approximate and exact nearest neighbour are effectively the same result for a user.[5]

At N = 1,000,000: HNSW needs ~20 effective operations. Grover with ideal qRAM needs ~785 oracle calls.[1][5] HNSW wins by ~40x, on a laptop, today. Grover's niche -- exact search at large N -- barely exists in real ML applications.

Why Grover's speedup still matters

  • One of the only proven, unconditional quantum speedups that has not been dequantized.
  • Empirically verifying it on a simulator is a concrete contribution -- literature is mostly theoretical.
  • The platform is architecturally ready for a qRAM backend if hardware ever exists.

Expected Final Outcomes

Hypotheses, contribution, and defensible claims

Expected findings (hypotheses)

  • Grover oracle count will follow the floor(pi * sqrt(N) / 4) curve empirically across N = 100 to 2000.
  • Swap test will converge to classical MRR at a measurable shot budget (hypothesis: 512-1024 shots).
  • Circuit depth and qubit count will grow predictably with dimension.
