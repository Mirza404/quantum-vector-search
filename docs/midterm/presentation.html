Context and Motivation

Why this problem matters

The search problem

  • Modern retrieval systems work in vector space -- text, images, audio all reduced to high-dimensional embeddings.
  • Cross-modal search (text query finding images) requires a shared embedding space: different modalities, same geometry.
  • As datasets grow to hundreds of millions of vectors, brute-force scan stops being viable.

Why quantum is worth studying here

  • Grover's algorithm provides a proven, unconditional quadratic speedup for unstructured search -- one of very few such results in quantum computing.
  • Quantum hardware is advancing. Understanding what algorithms require today informs engineering decisions for tomorrow.
  • Most quantum search literature analyses circuit complexity theoretically. We run real Qiskit circuits on real embeddings and measure what the resources actually look like.

CLIP aligns text and image encoders in a shared vector space.[6]   Source: Radford et al., 2021 (OpenAI)

Problem Definition, Scope, and Objectives

What we are building and measuring

Research questions

  • Does the Grover oracle call count follow the expected O(sqrt(N)) curve empirically across dataset sizes?
  • Can the quantum swap test match classical MRR at a justifiable shot budget?
  • Is quantum vector search worth using even if the hardware problem (qRAM) were solved?

Objectives

  • Implement and verify quantum algorithms (swap test, Grover oracle) on a real circuit simulator.
  • Benchmark all engines on identical data with well-defined, fair KPIs.
  • Expose results through a web interface: user selects from predefined ground-truth queries, sees ranked results from all engines side by side.
  • Produce a rigorous, honest analysis of quantum search feasibility -- today and under future hardware assumptions.

Scope

In scope: Simulator-only benchmarks, CLIP embeddings on Flickr30k, web UI for comparison, academic analysis of quantum resource costs and algorithmic limits.
Possible future additions: Real quantum hardware (IBM free tier has reported multi-hour queue waits and significant noise beyond ~7 qubits, so not included in this phase). Production deployment. Larger datasets beyond the current benchmark configuration.

Why simulator and not IBM hardware

  • AerSimulator is noise-free and deterministic -- ideal for measuring oracle count scaling.
  • Shot noise exists identically on simulator and real hardware; our MRR results are directly transferable.
  • IBM free tier: reported multi-hour queue waits, significant noise beyond ~7 qubits -- would obscure the algorithmic signal we're measuring.

Theoretical Background

CLIP, vector search, and quantum fundamentals

CLIP -- shared vector space

  • Contrastive pre-training: paired text + image examples, aligned encoders.
  • Result: cosine similarity between a text vector and an image vector measures semantic match.
  • We use ViT-B/32 at 512 dimensions.[6]

Vector similarity search

  • Given a query vector, find the k nearest vectors in a dataset by distance or angle.
  • Classical approaches range from exact O(N) scans to approximate O(log N) graph traversal.
  • Quantum approaches aim to exploit superposition to reduce the number of comparisons needed.

Quantum computing foundations

  • Superposition: a qubit can be in a combination of |0⟩ and |1⟩ simultaneously, described by complex amplitudes. n qubits represent 2ⁿ states at once.
  • Amplitude encoding: a d-dimensional vector is loaded into log₂(d) qubits by assigning its components as quantum amplitudes -- exponential compression.
  • Interference: algorithms manipulate amplitudes so that wrong answers cancel out and the correct answer is amplified before measurement.
  • Measurement: collapses the superposition to a single outcome with probability equal to the amplitude squared. Algorithms must make the correct outcome dominant first.

Bloch sphere: any single-qubit state as a point on the unit sphere. Source: Wikipedia Commons

Engine 1: Brute Force Cosine

Classical baseline -- deterministic ground truth

How it works

  • No index, no approximation. Computes cosine similarity between the query and every vector in the dataset, returns top-k by score.
  • Cosine similarity: cos(θ) = q·d. All vectors are L2-normalised at ingest, so the denominator is 1 and this reduces to a dot product.
  • Complexity: O(N) per query. Exact and deterministic.
  • Role: ground truth. MRR for every other engine is computed against this output.
The only engine whose output requires no validation assumption. It defines correctness for all MRR calculations.

cos(θ) = a·b / (|a|·|b|). Small angle = high similarity = top rank. Source: Wikipedia Commons

Criteria: 2, 3

Engine 2: FAISS Flat L2

Classical, SIMD-accelerated -- production-grade exact search

How it works

  • FAISS (Facebook AI Similarity Search): production library used at scale. IndexFlatL2 stores all vectors uncompressed with a SIMD-vectorised (AVX2/AVX-512) inner loop.
  • L2 vs cosine: ||q − d||² = 2 − 2·(q·d) = 2(1 − cos θ). On L2-normalised vectors, minimising L2 distance is identical to maximising cosine similarity -- rankings match brute force exactly.
  • Complexity: O(N) per query, exact. Faster wall-clock time than brute force on large N due to hardware vectorisation.
  • Role: production classical reference -- what a real system would use for exact search today.

||q−d||² = 2 − 2·(q·d) = 2(1−cos θ). Minimising L2 = maximising cosine on normalised vectors. Source: Wikipedia Commons

Criteria: 2, 3

Engine 3: Qiskit Swap Test

Real quantum circuit -- vector similarity via measurement

Circuit

  • 1 ancilla + 2 registers (log₂(dim) qubits each). Measurement encodes vector overlap.
  • Repeat N times (once per candidate). Average shots for accuracy.
Similarity estimate, not a speedup. Complexity: O(N) circuits.

Resources and accuracy

  • Qubits: 2·log₂(dim) + 1. At dim=64 → 13 qubits; dim=128 → 15 qubits.
  • Shots: MRR converges to brute force at 512–1024 shots (verified).
  • Depth: shallow -- grows with dim due to amplitude encoding, not with N.
  • Simulator: Qiskit AerSimulator -- noiseless.

Role in the benchmark

  • Verifies the swap test produces the expected overlap estimate against theory.
  • Maps shots-to-quality curve: minimum shot budget for acceptable MRR.
  • Measures qubit count and circuit depth growth vs dimension.
Criteria: 2, 3, 4

Engine 4: Qiskit Grover Oracle

O(√N) oracle calls -- central scaling experiment

How Grover's algorithm works

Oracle marks the target; diffusion amplifies it. Source: Wikipedia Commons

  • State prep: all N vectors in superposition. Requires qRAM (see slide 15).
  • Oracle: flips phase of the target (most similar) state.
  • Diffusion: amplifies the target amplitude. Repeat ⌊π·√N / 4⌋ times.

What we benchmarked

  • Hardcoded toy state prep (qRAM does not exist). Oracle + diffusion are real Qiskit circuits on AerSimulator.
  • Central result: oracle call count follows floor(π·√N / 4) empirically (N = 100 to 2000).
O(√N) oracle scaling is real and proven.[7] We verify our circuit reproduces it.

Future Extension: HNSW Engine

The production standard -- natural next benchmark engine

Sparse top layers for fast navigation, dense bottom for precision. Query greedy-descends. O(log N) total. Source: Wikipedia Commons

How HNSW works

  • Multi-layer proximity graph. Top layers: sparse, long-range. Bottom layer: dense, precise.
  • Query traversal: O(log N) edge hops. Recall: 95–99%+ in practice.[5]
  • Already in use: pgvector's HNSW index backs our PostgreSQL storage layer.

Why add it as an engine

  • Completes the comparison: O(N) brute force → O(log N) HNSW → O(√N) Grover [ideal qRAM].
  • Strategy pattern: only needs build_index() + search().
At N = 1,000,000: HNSW ~20 ops. Grover + ideal qRAM ~785 oracle calls.[1][5] HNSW wins by ~40x -- on a laptop, today.

The Honest Quantum Picture

What the theory and the numbers actually say

The two-step problem

  • Step 1 -- State preparation: Load all N vectors into superposition. Without qRAM this is O(N) gates -- same as classical scan.[4]
  • Step 2 -- Search: Grover finds the answer in O(sqrt(N)) oracle calls. This works and is verified by our circuits.
Without qRAM: O(N) loading + O(sqrt(N)) search = O(N) total. The speedup requires hardware that does not exist.

qRAM hardware reality

The bucket-brigade qRAM model requires O(N) quantum routing nodes as physical hardware.[4] With error correction (~1,000 physical qubits per logical qubit), even a small dataset demands hardware orders of magnitude beyond anything that exists or is on any roadmap:

At N = 1,000,000: HNSW needs ~20 effective operations. Grover with ideal qRAM needs ~785 oracle calls.[1][5] HNSW wins by ~40x, on a laptop, today. Grover's niche -- exact search at large N -- barely exists in real ML applications.

Why Grover's speedup still matters

  • One of the only proven, unconditional quantum speedups that has not been dequantized.
  • Empirically verifying it on a simulator is a concrete contribution -- literature is mostly theoretical.
  • The platform is architecturally ready for a qRAM backend if hardware ever exists.

Conclusion

What this project is -- and what it is not

What this project actually is

  • A rigorous, reproducible benchmarking platform for comparing quantum and classical vector search engines on identical data with well-defined KPIs.
  • An empirical verification that Grover's O(√N) oracle scaling holds in practice -- filling the gap between theoretical claims and measured circuit behaviour.
  • An honest, documented analysis of exactly where quantum search stands today and what hardware it would need to become practical.
The point is not to prove quantum is better. The point is to measure, with real circuits and real data, exactly why it is not -- and under what conditions it could be.

What the results show

  • Quantum swap test: produces correct similarity estimates at sufficient shot budget, but provides no speedup over classical. Cost: O(N) circuits.
  • Grover oracle: O(√N) oracle count is real and empirically verified. But the bottleneck is state preparation -- O(N) gates without qRAM that does not exist.
  • HNSW (classical): achieves O(log N) approximate search on a laptop today, with 95–99%+ recall. Even ideal qRAM would not close this gap.
Practical verdict: Quantum vector search is not useful in practice today, and will not become useful until fault-tolerant qRAM hardware exists -- which is not on any current roadmap. Classical approximate search (HNSW) dominates by every metric that matters to a real application.

Why this analysis matters

  • Quantum hype is common; rigorous negative results are rare and valuable.
  • This platform gives future researchers a reproducible baseline to revisit as hardware evolves.
