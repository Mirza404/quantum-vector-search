Context and Motivation

Why this problem matters

The search problem

  • Text, images, and audio can all be encoded as high-dimensional number vectors. Once everything is a vector, the same geometry applies: similarity between two items is the angle between their vectors.
  • This is why cross-modal search works: a text description and its matching image can be placed near each other in the same vector space. Finding the nearest vector finds the match -- regardless of modality.
  • As datasets grow to hundreds of millions of vectors, checking every one becomes too slow. Smarter algorithms are needed.

Why quantum is worth studying here

  • Grover's algorithm provides a proven, unconditional quadratic speedup for exact unstructured search.[1]
  • Quantum hardware is advancing. Understanding what algorithms require today informs engineering decisions for the future.
  • Grover's speedup is well-established theoretically. This project runs real Qiskit circuits on real embeddings to measure what the resource costs actually look like in practice.

CLIP aligns text and image encoders in a shared vector space.[6]   Source: Radford et al., 2021 (OpenAI)

Problem Definition, Scope, and Objectives

What we are building and measuring

Research questions

  • Does the Grover oracle call count follow the expected O(sqrt(N)) curve empirically across dataset sizes?
  • Can the quantum swap test match classical MRR at a justifiable shot budget?
  • Is quantum vector search worth using even if the hardware problem (qRAM) were solved?

Objectives

  • Implement and verify quantum algorithms (swap test, Grover oracle) on a real circuit simulator.
  • Benchmark all engines on identical data with well-defined, fair KPIs.
  • Expose results through a web interface: user selects from predefined ground-truth queries, sees ranked results from all engines side by side.
  • Produce a rigorous, honest analysis of quantum search feasibility -- today and under future hardware assumptions.

Scope

In scope: Simulator-only benchmarks, CLIP embeddings on Flickr30k, web UI for comparison, academic analysis of quantum resource costs and algorithmic limits.

Possible future addition

Real IBM quantum hardware. Not included in this phase -- the free tier is reported to have long queue waits and significant gate noise beyond ~7 qubits, which would make controlled scaling experiments hard to interpret cleanly. AerSimulator gives the same shot noise with none of the interference.

Theoretical Background

CLIP, vector search, and quantum fundamentals

CLIP -- shared vector space

  • Contrastive pre-training: paired text + image examples, aligned encoders.
  • Result: cosine similarity between a text vector and an image vector measures semantic match.
  • We use ViT-B/32 at 512 dimensions.[6]

Vector similarity search

  • Given a query vector, find the k nearest vectors in a dataset by distance or angle.
  • Classical approaches range from exact O(N) scans to approximate O(log N) graph traversal.
  • Quantum approaches aim to exploit superposition to reduce the number of comparisons needed.

Quantum computing foundations

  • Superposition: n qubits can represent all 2n states simultaneously. Think of a maze where you send a copy of yourself down every path at once -- classically you try one path at a time.
  • Amplitude encoding: a d-dimensional vector is loaded into log₂(d) qubits by assigning its components as quantum amplitudes -- exponential compression.
  • Interference: algorithms manipulate amplitudes so that wrong answers cancel out and the correct answer is amplified. In the maze analogy: all dead-end copies destructively interfere; only the exit survives.
  • Measurement: collapses superposition to a single outcome. The algorithm must make the correct outcome dominant before measuring.

Bloch sphere: any single-qubit state as a point on the unit sphere. Source: Wikipedia Commons

The Honest Quantum Picture

What the theory and the numbers actually say

The two-step problem

  • Step 1 -- State preparation: Load all N vectors into superposition. Without qRAM this is O(N) gates -- same as classical scan.[4]
  • Step 2 -- Search: Grover finds the answer in O(sqrt(N)) oracle calls. This works and is verified by our circuits.
Without qRAM: O(N) loading + O(sqrt(N)) search = O(N) total. The speedup requires hardware that does not exist.

qRAM hardware reality

Think of it like a classical computer: the qubit is the processor -- it runs the algorithm. qRAM is the storage -- it holds the dataset and feeds it to the processor. Both are built from qubits, but qubits are fragile: reliable storage needs ~1,000 physical qubits wrapping each memory slot for error correction. qRAM needs one slot per vector, so the storage cost is N vectors × 1,000 qubits. The algorithm itself only needs log₂(N) qubits to index N items -- just enough to represent a number up to N.[4]

Dataset sizeClassical RAMAlgorithm qubits (log₂N)Storage qubits (qRAM)
1,000 vectors~2 MB~10~1,000,000
1,000,000 vectors~2 GB~20~1,000,000,000

qRAM hardware does not exist on any roadmap.

Three compounding reasons Grover loses

#ProblemStatus
1qRAM doesn't exist -- state prep is O(N), speedup cancelled[10]Hard blocker today
2Even if qRAM existed, it requires O(N) quantum nodes -- same linear scaling as classical RAM, but at astronomical cost per unit. 1B vectors = ~1 trillion physical qubits of memory.[11]Economic blocker at scale
3Even ignoring 1 and 2, Grover's O(sqrt(N)) is beaten by HNSW's O(log N) classically. At N=1M: HNSW ~20 ops, Grover ~785 oracle calls.[5]Algorithmic blocker
HNSW needs O(N) classical bits (cheap RAM) + O(log N) search ops. Grover + qRAM needs O(N) quantum routing nodes (astronomically more expensive per unit) + O(sqrt(N)) search ops -- and O(sqrt(N)) > O(log N). Classical wins on every axis.

Why Grover's speedup still matters

  • Built for unstructured search -- no index, no geometry required. Works on any computable condition, not just vector distance.
  • Proven optimal: there is a lower bound showing no quantum algorithm can beat O(√N) for unstructured search. Grover hits it exactly -- provably the best possible, not just the best known.[7]
  • A building block inside larger quantum algorithms and a long-term cryptographic threat -- its value is abstract and future-facing, not in vector search today.

Conclusion

What this project is -- and what it is not

What this project actually is

  • A rigorous, reproducible benchmarking platform for comparing quantum and classical vector search engines on identical data with well-defined KPIs.
  • An empirical verification that Grover's O(sqrt(N)) oracle scaling holds in practice -- measuring theory against real circuit behaviour.
  • An honest, documented analysis of exactly where quantum search stands today and what hardware it would need to become practical.
The point is not to prove quantum is better. The point is to measure, with real circuits and real data, exactly why it is not -- and under what conditions it could be.

What the results show

  • Quantum swap test: produces correct similarity estimates at sufficient shot budget, but provides no speedup over classical. Cost: O(N) circuits.
  • Grover oracle: O(sqrt(N)) oracle count is real and empirically verified. But the bottleneck is state preparation -- O(N) gates without qRAM that does not exist.
  • HNSW (classical): achieves O(log N) approximate search on a laptop today, with 95-99%+ recall. Even ideal qRAM would not make Grover competitive.
"Quantum computers will be practical for 'big compute' problems on small data, not big data problems." [10]
Vector search is a data volume problem, not a computation problem -- and loading that volume into superposition is exactly the qRAM blocker above. Quantum wins on small data with massive computation (molecule simulation, cryptography), not dataset scans.

Why this analysis matters

  • Quantum hype is common; rigorous negative results are rare and valuable.
  • This platform gives future researchers a reproducible baseline to revisit as hardware evolves.
