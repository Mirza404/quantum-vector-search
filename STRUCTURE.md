# Project Map: Quantum Vector Search

This note walks every folder and file in the repo and describes, in plain language, what it already does and what it still needs in the future. Think of "strategy" simply as "a plug slot"—each strategy file defines the expected behavior, while the concrete files show one possible plug that can be swapped later.

## Root directory (`quantum-vector-search/`)
- `.git/` – version-control history and internal plumbing created by Git; nothing to edit manually.
- `README.md` – human overview of the current prototype, including how the strategies connect and which scripts to run.
- `STRUCTURE.md` – this map.
- `backend/` – all executable Python code, sample data, scripts, and virtual-env helpers live here.
- `docs/` – copies of the architecture and benchmarking master plans; they are reference material for future milestones.

## `docs/`
- `ARCHITECTURE.md` – the global blueprint describing the five project phases (repository layer, embeddings, engines, benchmarking, API/UI).
- `BENCHMARKING_STRATEGY.md` – detailed instructions for how the automated study must behave (ground-truth set, resumable runs, required metrics, clean-slate rule).

## `backend/` (Python project root)
- `.gitignore` – tells Git to ignore generated folders like caches, artifacts, and compiled files.
- `.python-version` – stores the Python version for tools like `pyenv`.
- `.venv/` – optional local virtual environment so dependencies install without touching the system Python; nothing in here needs reviewing.
- `artifacts/benchmarks/` – output folder that the benchmarking script writes (`results.csv`, `Report.md`). Safe to delete and regenerate.
- `data/sample_dataset/` – tiny fixed dataset used for tests.
  - `metadata.csv` – CSV table with `id`, human text, and relative image path.
  - `images/*.jpg` – placeholder image files so paths resolve; they do not hold real pixels yet.
  - `ground_truth.json` – list of benchmark queries (text + the ID that should be the best answer).
  - `cache/` – embeddings cache (`embeddings.npy` + `manifest.json`) created by `scripts/build_embeddings.py`; delete when data changes.
- `main.py` – console demo. Loads the dataset via the data-loader strategy, ensures embeddings exist (uses cache if available), embeds the first ground-truth query, and prints results from both engines. No web server yet.
- `pyproject.toml` – Python project metadata plus dependency list (FastAPI, FAISS, NumPy, Pytest, Ruff). Even though FastAPI/FAISS are listed, the current code only uses mocks; the real engines/server are still future work.
- `uv.lock` – dependency lockfile generated by the `uv` package manager; keeps reproducible versions.

### `backend/scripts/`
- `build_embeddings.py` – command-line tool that turns every dataset text into a numeric vector. It remembers previous runs via text hashes so unchanged rows are reused. Today it uses a mock embedder but can later call CLIP or other models.
- `run_benchmarks.py` – command-line harness that (1) loads embeddings, (2) loops over all queries and requested vector dimensions, (3) runs each search engine, (4) times them, (5) scores accuracy against `ground_truth.json`, and (6) appends results to CSV/Markdown. It already enforces the resumable + append-only rules from the benchmarking plan.

### `backend/src/`
- `quantum_vector_search.egg-info/` – auto-generated packaging metadata; safe to ignore (Git already does).
- `qvs/` – main Python package; every subfolder here represents a strategy “slot”.
  - `__init__.py` – marks the package; currently empty.

#### `qvs/repository/`
- `base.py` – defines `DatasetRecord`, `Dataset`, and `BaseDataLoader`. Think of this as “what data loaders must provide” (`get_dataset`, `describe_source`).
- `local_csv.py` – concrete plug: reads the CSV + image paths from `data/sample_dataset/` and returns a `Dataset` object.
- `__init__.py` – exports the public names so the rest of the code can import `from qvs.repository import LocalCSVDataLoader`.
*Future work:* add other loaders (e.g., SQL, cloud storage) by subclassing `BaseDataLoader` without touching callers.

#### `qvs/pipeline/`
- `embedding.py` – declares the `EmbeddingGenerator` interface and provides `MockCLIPEmbeddingGenerator`, a deterministic pseudo-embedder for now.
- `cache.py` – simple cache manager that saves embeddings to `.npy` and tracks per-row hashes so reruns only regenerate modified text.
- `__init__.py` – exports the cache + generator types.
*Future work:* swap `MockCLIPEmbeddingGenerator` for a real multimodal embedder, keep the same cache API.

#### `qvs/engines/`
- `__init__.py` – defines `SearchEngineStrategy` (common API: `name`, `build_index`, `search`) and the `SearchResult` data holder.
- `vector_mock.py` – classical baseline that scores by negative Euclidean distance (no external libraries required).
- `quantum_mock.py` – toy quantum-like scorer that normalizes vectors, simulates overlap probability, and adds configurable noise via `shots` and `layers` parameters.
*Future work:* implement real `FAISSEngine` and `QiskitEngine` by following the same interface.

#### `qvs/benchmark/`
- `models.py` – plain data classes: `BenchmarkQuery` (text + target) and `BenchmarkResult` (stores timing, accuracy, and metadata per run). Provides helper to load the JSON query list.
- `storage.py` – storage strategy interface plus the CSV+Markdown implementation that appends rows and regenerates `Report.md` with summary tables.
- `__init__.py` – exports the benchmark helpers for easy importing.
*Future work:* add a database-backed storage strategy if/when needed.

### `backend/.venv/` (optional)
- Contains Python binaries, libraries, and scripts installed locally so system packages remain untouched. You only need it to run commands; no project logic here.

## What you already have vs. what’s missing
- **Already in place:** sample dataset, repository abstraction, embedding cache & mock embedder, mock search engines under a shared interface, benchmark data classes + storage, automated benchmarking loop, and a README describing how to run it all.
- **Still to implement (per the architecture docs):** real embedding model (CLIP or similar) instead of mocks, real FAISS + Qiskit engines, a FastAPI backend that exposes `/search` and `/api/benchmarks`, persistent storage beyond CSV (if needed), and eventually the React dashboard. Those future components can plug into the existing interfaces without reorganizing today’s code.

Use this map as a checklist: every time you add a new strategy implementation, create it inside the relevant `qvs/*` folder and register it via the interface, then wire it up in scripts or the future API layer.
