\section{System Overview}
\label{sec:method:overview}

% One paragraph describing the modular monolith - data loader, embedding
% pipeline, engines, storage, API+UI. Reference Figure \ref{fig:architecture}.

The system is a modular monolith for cross-modal retrieval. A text query and the
dataset images are embedded into the same vector space, then one interchangeable
engine ranks the image vectors closest to the query vector. Classical, quantum,
and hybrid engines all implement the same \texttt{SearchEngineStrategy}
interface, with \texttt{build\_index} and \texttt{search} as the required
operations. The rest of the harness, data loading, embedding, query execution,
and result storage, is therefore identical for every engine.\footnote{The full
source code, benchmark harness, and dataset configuration are publicly available
at \url{https://github.com/Mirza404/quantum-vector-search}.}

Figure~\ref{fig:architecture} shows the five phases. A
\texttt{DirectoryDataLoader} reads the images and ground-truth query pairs; CLIP
embeds the images and text; the selected engine builds an index and answers the
query; PostgreSQL records accuracy and resource measurements; FastAPI and React
surface the rankings. The experiment is configured in \texttt{benchmarks.yaml},
where the active engines, dimensions, and shot counts are selected without
editing source code.

The experiment follows the script sequence in Figure~\ref{fig:methodology}. The
import script downloads the fixed Flickr30k subset and writes the ground-truth
query file. The indexing script encodes each image with CLIP and stores one
vector per image in PostgreSQL. The benchmark script then iterates over every
enabled engine, vector dimension, shot count, and query. Quantum and hybrid
engines run on Qiskit's noiseless simulator, so the sweep measures algorithm
behaviour and finite-shot sampling rather than device noise.

Each run is stored as one row in \texttt{benchmark\_results}, including MRR,
oracle calls, circuit depth, and qubit count where applicable. The IBM
validation script executes the hybrid swap test once on real hardware at two
dimensions, two candidates, and 32 shots. This result is tagged with the IBM
engine ID and is treated as a validation run, not as part of the normal sweep.
Finally, the report-generation script produces \texttt{benchmark\_report.md}
directly from the database; Chapter~\ref{ch:results} uses that generated file as
the source of truth for all reported numbers.

\section{Dataset and Ground Truth}
\label{sec:method:dataset}

The evaluation set is a fixed 20-image subset of Flickr30k, an image-caption
corpus in which each photograph has several human-written captions. The importer
requests a fixed configuration and split, then reads the first twenty rows in the
dataset viewer order, with no random sampling or shuffling. A different machine
therefore regenerates the same images and queries. Twenty images is small for
information retrieval, but it keeps the full seven-engine simulator sweep
tractable on a CPU; the separate IBM hardware validation is excluded from this
count because it is a tiny real-device sanity check.

For each image we use the first caption as the query text and record the image's
Flickr identifier as the target. The result is a one-to-one evaluation: every
query has exactly one correct image. This choice ignores alternate captions and
cannot reward a different image that a human might also consider relevant, but it
makes scoring deterministic. An engine ranks all twenty images, and MRR assigns
full credit when the target is first and partial credit when it appears lower.

\section{Embedding Pipeline}
\label{sec:method:embeddings}

CLIP embeds both images and text into one vision-language space
\cite{radford2021clip}. It is trained contrastively so matching images and
captions are close while mismatches are pushed apart \cite{radford2021clip}. We
use ViT-B/32, which produces 512-dimensional vectors, and L2-normalise every
image and query vector. On unit vectors, cosine similarity, Euclidean distance,
and the squared inner product estimated by the swap test induce the same
ranking, which lets MRR be compared across engines that compute different
similarity quantities.

The benchmark uses truncated dimensions of 64, 128, and 256 rather than the full
512. The reason is simulation cost. Amplitude encoding a $d$-dimensional vector
requires $\lceil \log_2 d \rceil$ qubits, and the swap test needs two such
registers plus one ancilla, or $1 + 2\lceil \log_2 d \rceil$ qubits. At 512
dimensions the swap test would need nineteen qubits and more than half a million
state-vector amplitudes per comparison; at 64 dimensions it needs thirteen
qubits and roughly eight thousand amplitudes. Truncation reduces simulator cost
but discards information, so the Results chapter measures the accuracy loss.
Engines that rely on unit length re-normalise after truncation.

\section{Search Engines}
\label{sec:method:engines}

\subsection{Classical Baselines}
\subsubsection{Brute-Force Cosine}
The reference engine compares the query with every image. After normalisation,
cosine similarity is the dot product, so the engine computes twenty dot products,
sorts the scores, and returns the top results. Its work grows as $O(N)$, but it
cannot skip the closest image. We use its ordering as the reference ranking for
the other engines.

\subsubsection{FAISS Flat (Exact)}
FAISS Flat delegates the same exhaustive search to FAISS, a similarity-search
library covering exact, approximate, and compressed-domain search
\cite{johnson2019faiss}. We use its exact L2 index. For unit vectors, L2 distance
and cosine similarity are linked by $\|a - b\|^2 = 2 - 2\cos\theta$, so the
ranking matches brute-force cosine after re-normalisation. FAISS Flat is
therefore a production-grade exact baseline that should reproduce the reference
ranking.

\subsubsection{FAISS HNSW (Approximate)}
FAISS HNSW uses a Hierarchical Navigable Small World graph, an approximate
nearest-neighbour index that searches a layered proximity graph instead of the
whole collection \cite{malkov2020hnsw}. Its expected query cost is
$O(\log N)$, with a recall trade-off at large scale. On our 20-image collection
the graph is too small for the approximation to matter, so HNSW reproduces the
exact ranking. Its role is to place the quantum engines next to the classical
index a real retrieval system would use.

\subsection{Quantum Engines}
\subsubsection{Qiskit Swap Test}
% Include the formula here. Equations must be numbered per FENS template.
The swap test estimates the squared inner product of two amplitude-encoded
vectors $\ket{\psi}$ and $\ket{\phi}$ via a single ancilla measurement:
\begin{equation}
P(\text{ancilla}=0) = \frac{1 + |\langle\psi|\phi\rangle|^2}{2}.
\label{eq:swaptest}
\end{equation}

The procedure originates in quantum fingerprinting, where it distinguishes two
unknown quantum states with high probability \cite{buhrman2001swaptest}. In our
engine each vector is amplitude-encoded into a register, and one ancilla qubit
controls a swap between the registers. Figure~\ref{fig:swaptest_circuit} shows a
four-dimensional example. Equation~\ref{eq:swaptest} converts the probability of
measuring the ancilla as zero into an overlap score. Because that probability is
estimated from a finite number of shots, the score has sampling error, so we
sweep the shot count. The swap test compares one query with one candidate; a
full ranking still needs $O(N)$ circuits. We run the sweep on Qiskit's Aer
simulator \cite{qiskit2024}, with a separate small IBM hardware validation.

\begin{figure}[ht]
  \centering
  \includegraphics[width=0.7\linewidth]{figures/swap_test_circuit.png}
  \caption[Swap-test circuit]{Swap-test circuit for a four-dimensional example,
  in which each vector is amplitude-encoded into two qubits. The ancilla
  ($q_0$) is placed in superposition, controls a swap of the two encoded
  registers ($q_1 q_2$ and $q_3 q_4$), and is measured after a second Hadamard.
  The probability of reading the ancilla as zero gives the squared overlap of
  the two states, as in Equation~\ref{eq:swaptest}.}
  \label{fig:swaptest_circuit}
\end{figure}

\subsubsection{Qiskit Grover (Hardcoded Oracle)}
Grover's algorithm finds a marked item in an unstructured collection using about
$\sqrt{N}$ oracle queries, a quadratic improvement over a classical scan
\cite{grover1996}. Our engine pads $N$ to a power of two, prepares a uniform
index superposition, then repeats an oracle and diffusion operator
$\lfloor\pi\sqrt{N}/4\rfloor$ times before measuring. The oracle marks the target
index, and the diffusion step amplifies it; Figure~\ref{fig:grover_circuit}
shows one iteration for four items.

The limitation is the oracle. A real vector-search oracle would need to decide,
in superposition, which stored vector is closest to the query, which requires
quantum random access memory (qRAM) for loading data in superposition
\cite{giovannetti2008qram}. Practical qRAM does not exist, and loading vectors
one by one costs $O(N)$, removing the advantage \cite{aaronson2015readfine}. We
therefore hardcode the nearest-neighbour index classically and test only the
Grover search step. The claim being tested is oracle-call scaling, not
end-to-end quantum speedup.

  \centering
  \includegraphics[width=\linewidth]{figures/grover_circuit.png}
  \caption[Grover circuit, one iteration]{One Grover iteration for a
  four-element index ($N=4$, two index qubits). After preparing a uniform
  superposition, the oracle flips the phase of the marked index and the
  diffusion operator amplifies its amplitude; the two qubits are then measured.
  The barriers separate the preparation, oracle, and diffusion phases.}
  \label{fig:grover_circuit}
\end{figure}

\subsubsection{Qiskit Grover with Quantum State Preparation}
The quantum-preparation Grover variant chooses the target index with quantum
measurements instead of a classical dot product. Each stored vector and query are
loaded with Qiskit state-preparation gates; a swap test estimates each overlap;
the highest measured overlap becomes the index marked by the Grover oracle. The
remaining Grover circuit is unchanged. This makes target selection noisy:
state-preparation circuits are deep, the swap tests have shot noise, and Grover
amplifies whichever index it is given. The Results chapter shows that this
variant is the least accurate engine.

\subsubsection{Hybrid HNSW + Swap Test}
The hybrid engine uses HNSW to shortlist candidates, then applies the swap test
only to that fixed-size pool. The classical stage performs the search; the
quantum stage reranks the survivors. This mirrors common retrieval pipelines in
which a fast approximate index proposes candidates and a slower scorer refines
them. Because the swap test is not a search algorithm, this is the most plausible
place for it in the system. On our small dataset the shortlist usually already
contains the target, so the hybrid tracks the classical baseline closely.

\subsubsection{IBM Hardware Validation}
To check that the quantum path is not only a simulator-only result, we ran the
hybrid engine once through IBM Quantum runtime. The run uses two-dimensional
vectors, two candidates, and thirty-two shots, small enough for the free-tier
device time and shallow enough for noisy hardware. It is a sanity check, not a
scaling experiment. The Results chapter reports low accuracy, as expected from
aggressive truncation, few shots, and real-device noise. We draw no scaling
conclusions from it.

\section{Evaluation Metrics}
\label{sec:method:metrics}

% Define MRR formally, then define oracle calls. Explain why wall-clock is per-
% engine only - the simulator-time issue.

The primary accuracy measure is mean reciprocal rank (MRR). For each query, the
engine receives the reciprocal of the rank at which the correct image appears:
rank one scores one, rank two scores one half, and a target outside the top $k$
results, with $k=10$, scores zero. Averaging over all queries gives
Equation~\ref{eq:mrr}. Because every engine is scored against the same target
image, MRR is comparable across engines.

\begin{equation}
\text{MRR} = \frac{1}{|Q|} \sum_{q \in Q} \frac{1}{\text{rank}_q},
\label{eq:mrr}
\end{equation}
where $\text{rank}_q$ is the position of the first relevant result for query
$q \in Q$ and $1/\text{rank}_q \coloneqq 0$ if no relevant result is in the top-$k$.

We also store operation count, the number of core comparisons, graph steps, or
oracle calls per query. Brute force, FAISS flat, and the standalone swap test
scale as $O(N)$; HNSW as $O(\log N)$; the hybrid as $O(\log N + M)$; and Grover
as $O(\sqrt{N})$ with $\lfloor\pi\sqrt{N}/4\rfloor$ oracle cycles. This count is
not tied to any particular hardware, so it is the only speed-like quantity
compared across engines.

For circuit engines we record depth and qubit count. Depth is a proxy for noise
exposure on real hardware, while qubit count is the circuit width. We do not
compare wall-clock time across engines because the quantum engines run on a
classical simulator whose runtime reflects state-vector simulation overhead.
Wall-clock appears only within an engine, where the hardware and simulator are
held fixed.

\section{Experimental Configuration}
\label{sec:method:config}

Table~\ref{tab:config} lists the sweep. Each engine is evaluated at every vector
dimension; quantum and hybrid engines also run at every shot count. With twenty
queries, this gives 900 simulator runs: three classical configurations and
twelve quantum configurations per dimension, across three dimensions. The IBM
hardware validation adds 20 rows, for a stored total of 920 results. The layers
parameter is fixed at one because the harness requires it, although no enabled
engine uses it. Results are stored in PostgreSQL and dumped to the committed seed
file.
